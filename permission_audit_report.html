Permission Audit Report

Flutter To-Do App — Calendar & Notification Permissions
Date: May 2026 Platform: Android & iOS Packages: device_calendar 4.3.3  |  flutter_local_notifications 17.2.2  |  permission_handler 12.0.1

1. Executive Summary

The app's permission infrastructure follows the correct conceptual pipeline (check → request → handle denial → redirect to Settings). Most of the core building blocks exist. However, several gaps — a broken Settings redirect, no rationale dialogs, and eager notification prompts — put the app below industry standard and will negatively affect grant rates and App Store review.

Overall verdict: Partially compliant. Functional for happy-path users, but denial recovery is partially broken and no rationale context is shown to users.

3. Calendar Permissions

3.1 How It Works Today

Calendar permission is managed entirely through the device_calendar plugin inside LocalCalendarService.getCalendars(). The check-then-request pattern is correct: 

var permissionsGranted = await _deviceCalendarPlugin.hasPermissions();
if (!permissionsGranted.isSuccess || permissionsGranted.data == false) {
  permissionsGranted = await _deviceCalendarPlugin.requestPermissions();
}

if (permissionsGranted.isSuccess && permissionsGranted.data == true) {
  final calendarsResult = await _deviceCalendarPlugin.retrieveCalendars();
  return calendarsResult.data ?? [];
} else {
  print('**Calendar permissions not granted**'); // silent – user sees nothing
}
Good: Permission is checked lazily (only when the calendar screen is opened), not at app launch. This is the correct deferred pattern.

3.2 Issues Found

No rationale dialog. The OS prompt appears with no prior context. Users who tap "Deny" can never be prompted again via code.
Silent failure. When denied, getCalendars() returns an empty list and prints to the console. The user sees an empty calendar picker with no explanation — they cannot tell if the list is empty by choice or because access was blocked.
No permanent-denial recovery for calendar. Unlike notifications, there is no code path that detects a permanently-denied calendar permission and redirects to device Settings.

4. Notification Permissions

4.1 How It Works Today

Notification permissions are initialised in main.dart at cold launch and handled by NotificationService. The flow covers both Android and iOS: 

// Android
final bool? result = await _notifications
    .resolvePlatformSpecificImplementation<
      AndroidFlutterLocalNotificationsPlugin>()
    ?.requestNotificationsPermission();

// Also requests exact alarm permission (Android 12+)
final bool? exactAlarmGranted = await _notifications
    .resolvePlatformSpecificImplementation<
      AndroidFlutterLocalNotificationsPlugin>()
    ?.requestExactAlarmsPermission();

// iOS
final bool? result = await _notifications
    .resolvePlatformSpecificImplementation<
      IOSFlutterLocalNotificationsPlugin>()
    ?.requestPermissions(alert: true, badge: true, sound: true);

A permission guard is also present before scheduling:

final bool hasPermission = await isNotificationPermissionGranted();
if (!hasPermission) {
  final bool granted = await requestNotificationPermission();
  if (!granted) {
    print("❌ Permission denied. Notification not scheduled.");
    return; // graceful fallback – correct
  }
}
Good: Exact alarm permission (SCHEDULE_EXACT_ALARM) is handled separately for Android 12+. Many apps miss this and silently deliver inexact alarms. Boot receiver is also configured so notifications survive restarts.

4.2 Issues Found

Bug – Settings redirect is broken. request_permissions.dart calls openAppSettings() for permanently-denied notifications, but the function is never imported. The call silently does nothing, leaving users with no recovery path.
Eager permission request. NotificationService.init() fires in main.dart before the user has seen any value from the app. Studies show cold-launch permission prompts are denied 40-60% more often than context-aware prompts. This should be deferred to the first time the user sets a reminder on a task.
No rationale dialog for notifications. Same issue as calendar — the OS prompt fires with no preceding explanation.
Missing iOS Info.plist entry for notifications. NSCalendarsUsageDescription exists but there is no notification usage description. While iOS 10+ does not strictly require a plist key for notifications (the system prompt handles it), Apple App Store review guidelines expect a description for every sensitive capability the app uses.

5. Denial Recovery Architecture

permission_handler.dart defines an enum and provider that correctly models the three states Android exposes: 

enum EnNotificationPermission { granted, denied, permanentlyDenied, unknown }

request_permissions.dart branches on this enum: 

if (_permissionHandler.notificationPermission == EnNotificationPermission.denied) {
  await NotificationService.requestNotificationPermission();
} else if (_permissionHandler.notificationPermission ==
           EnNotificationPermission.permanentlyDenied) {
  openAppSettings();  // ← import missing – this line does nothing
}
Architecture is correct. The three-state model (granted / denied / permanentlyDenied) is exactly what industry guides recommend. The only issue is the missing import that makes the openAppSettings() call a no-op. Adding import 'package:permission_handler/permission_handler.dart'; would fix it.

7. Industry-Standard Permission Flow (Reference)

  1. Defer the ask — only request permission at the moment the user initiates an action that needs it (e.g. tapping "Add to Calendar", setting a reminder).
  2. Show rationale first — display your own dialog explaining why the permission is needed and what value it unlocks. Provide "Not Now" and "Continue" options.
  3. Trigger OS prompt — only after the user taps "Continue".
  4. Handle denied — silently fall back. Do not ask again in the same session. Show a non-blocking banner if the feature is now unavailable.
  5. Handle permanently denied — detect permanentlyDenied, show a dialog explaining the situation, offer a "Go to Settings" button that calls openAppSettings().
  6. Re-check on resume — listen to AppLifecycleState.resumed and re-check permission status in case the user granted it from Settings.
Steps 1, 3, and 4 are already implemented. Steps 2, 5, and 6 are the primary gaps.
