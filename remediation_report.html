3. Detailed Changes

Critical F-01 — MSAL Client ID Mismatch

outlook_sign.dart:5–6

The hardcoded client ID in OutlookAuthService did not match the client_id in msal_config.json. MSAL validates the redirect URI against the config file, so the mismatch caused authentication to target the wrong Azure AD app registration. The Dart constant was updated to match the JSON config. 

// Before
static const _clientId = '6450d522-3a1c-4005-ae93-1fdc7f91aea2';

// After — now matches msal_config.json client_id
static const _clientId = '1046c7a4-3a52-4845-b477-81163b44f47e';

Bug B-01 — Stale Token Written to Storage in initialize()

outlook_sign.dart — initialize()

The storage write in initialize() used _accessToken (the previous/null value) instead of silentToken (the freshly acquired value), because the assignment happened after the write. The order was swapped so the fresh token is always persisted. 

// Before (buggy) — writes old _accessToken, then assigns silentToken
if (silentToken != null) {
  await storage.write(key: 'outlook_cal_accessToken', value: _accessToken);
  _accessToken = silentToken;
  return true;
}

// After (fixed) — assign first, then write the fresh value
if (silentToken != null) {
  _accessToken = silentToken;
  await storage.write(key: 'outlook_cal_accessToken', value: silentToken);
  return true;
}

Bug B-02 — Expired Token Left in Memory After Failed restoreLastSession()

outlook_sign.dart — restoreLastSession()

_accessToken was assigned the saved (potentially expired) token before calling acquireTokenSilently(). If silent verification failed, _accessToken remained set to the stale value — causing all subsequent API calls to use an expired token with no clear error reason. The premature assignment was removed; _accessToken is now only set after successful verification, and explicitly cleared to null on failure. 

// Before (buggy) — sets _accessToken before verification
_accessToken = savedToken;
final silentToken = await acquireTokenSilently();
if (silentToken != null) { _accessToken = silentToken; return true; }
return false; // _accessToken still holds stale savedToken!

// After (fixed) — only assigns on success, clears on failure
final silentToken = await acquireTokenSilently();
if (silentToken != null) { _accessToken = silentToken; return true; }
_accessToken = null;
return false;

High F-04 — Access Tokens Removed from Console Logs

outlook_sign.dart — acquireTokenSilently() · signIn()

Two print() statements were emitting full OAuth access tokens to the debug console. Both were replaced with token-safe messages that confirm success without exposing the token value. 

// acquireTokenSilently() — Before
print("🔑 Silent token acquired: ${result.accessToken}");

// acquireTokenSilently() — After
print("🔑 Silent token acquired");

// signIn() — Before
print('Access Token: ${result.accessToken}');

// signIn() — After
print('✅ Interactive sign-in succeeded');

High F-02 — Forced Interactive Sign-In at Cold Start Removed

google_sign.dart — initApp() · outlook_sign.dart — initialize()

Both Google and Outlook were triggering interactive sign-in dialogs at app startup before the user had requested any calendar feature. This violated OAuth 2.0 incremental authorization and blocked users who only wanted the task manager. Both startup methods now only attempt silent restoration; interactive sign-in must be triggered by a deliberate user action (e.g. tapping "Connect Calendar"). 

// google_sign.dart initApp() — Before
GoogleSignInAccount? user = await signInSilently();
if (user == null) user = await signIn(); // ← forced interactive dialog
currentUser = user;

// google_sign.dart initApp() — After
currentUser = await signInSilently();

// outlook_sign.dart initialize() — Before
_accessToken = await signIn(); // ← interactive fallback on silent failure
if (_accessToken != null) { ... return true; }

// outlook_sign.dart initialize() — After (silent only)
print("⚠️ Silent sign-in failed — interactive sign-in required.");
return false;

Low F-09 — Outlook Sign-Out Now Deletes Stored Token

outlook_sign.dart — signOut()

signOut() cleared the in-memory _accessToken but left the token persisted in FlutterSecureStorage. On the next launch, restoreLastSession() would find and attempt to reuse this stale token. A storage.delete call was added alongside the existing in-memory clear. 

// Before — storage key left behind
await _pca.signOut();
_accessToken = null;

// After — storage cleaned up too
await _pca.signOut();
_accessToken = null;
await storage.delete(key: 'outlook_cal_accessToken');

Medium F-05 — AI Endpoint Now Authenticated

lib/services/AiTaskService.dart — generateTasks()

The Supabase Edge Function was called with only a Content-Type header and no authorization. Any caller knowing the URL could invoke it and incur LLM costs. The Supabase anon key is now attached as a Bearer token via AppConfig.supabaseAnonKey, which enables Supabase's JWT verification on the server side. 

// Before — no auth
headers: {'Content-Type': 'application/json'},

// After — anon key Bearer token attached
headers: {
  'Content-Type': 'application/json',
  'Authorization': 'Bearer ${AppConfig.supabaseAnonKey}',
},

Medium F-06 — Supabase anonKey Extracted to AppConfig

lib/config/app_config.dart (new) · lib/main.dart

The Supabase URL and anon key were hardcoded directly in main.dart. A new AppConfig class centralises both values and reads from --dart-define=SUPABASE_ANON_KEY at build time, falling back to the current literal only for local development. This stops the key from spreading to multiple files and documents the production override mechanism. 

// lib/config/app_config.dart (new file)
class AppConfig {
  static const supabaseUrl = 'https://vzltupelovhgagglqpjf.supabase.co';
  // flutter run --dart-define=SUPABASE_ANON_KEY=<key>
  static const supabaseAnonKey = String.fromEnvironment(
    'SUPABASE_ANON_KEY',
    defaultValue: 'eyJ...', // dev fallback only
  );
}

// main.dart — Before
await Supabase.initialize(
  url: "https://vzltupelovhgagglqpjf.supabase.co",
  anonKey: "eyJhbGci...",
);

// main.dart — After
await Supabase.initialize(
  url: AppConfig.supabaseUrl,
  anonKey: AppConfig.supabaseAnonKey,
);

Medium F-07 — AuthProvider Created (Centralized Auth State)

lib/providers/auth_provider.dart (new) · lib/main.dart

Auth state was fragmented across three independent locations (GoogleAuthService.currentUser, OutlookAuthService._accessToken, db.accountDetails) with no ChangeNotifier to propagate changes to the widget tree. A new AuthProvider was created as a single source of truth exposing isGoogleSignedIn, isOutlookSignedIn, and displayName. It is initialised with the actual post-startup auth state and registered in MultiProvider. 

// lib/providers/auth_provider.dart (new file)
class AuthProvider extends ChangeNotifier {
  bool _isGoogleSignedIn;
  bool _isOutlookSignedIn;
  String _displayName;

  bool get isGoogleSignedIn => _isGoogleSignedIn;
  bool get isOutlookSignedIn => _isOutlookSignedIn;
  String get displayName => _displayName;

  void setGoogleSignedIn(bool value, {String displayName = ''}) { ... }
  void setOutlookSignedIn(bool value) { ... }
  void signOutGoogle() { ... }
  void signOutOutlook() { ... }
}

// main.dart — registered after auth init
final authProvider = AuthProvider(
  isGoogleSignedIn: GoogleAuthService.currentUser != null,
  isOutlookSignedIn: OutlookAuthService.accessToken != null,
  displayName: GoogleAuthService.currentUser?.displayName ?? '',
);
// ... added to MultiProvider:
ChangeNotifierProvider.value(value: authProvider),

Medium F-08 — Google Calendar Scope Reduced to calendar.events

lib/services/google_sign.dart:11

The app was requesting the full auth/calendar scope which includes read/write access to all calendar settings and properties. Since the app only creates and edits events, the narrower calendar.events scope is sufficient. This reduces the permissions shown in the Google consent dialog and follows the principle of least privilege. 

// Before — full calendar access
'https://www.googleapis.com/auth/calendar',

// After — events only
'https://www.googleapis.com/auth/calendar.events',

Bug B-03 — HTTP Client Leak in _GoogleAuthClient Fixed

lib/services/google_sign.dart — _GoogleAuthClient

_GoogleAuthClient instantiated a new http.Client() on every call to _initApis() (which runs before every API use). Each discarded instance leaked an open connection pool. The _inner field was made static so a single shared client is reused across all instances. 

// Before — new Client() per instantiation
final http.Client _inner = http.Client();

// After — single shared client, no leak
static final http.Client _inner = http.Client();

4. Deferred Finding

Deferred F-03 — Route-Level Authorization Guards

main.dart:144–163

This finding was explicitly deferred by the developer. All seven application routes remain accessible without an authentication check. The AuthProvider created for F-07 provides the necessary state when this is eventually implemented — widgets can call Provider.of<AuthProvider>(context).isGoogleSignedIn to gate access to calendar pages, or the app can migrate to go_router with redirect callbacks.
