01 — Critical Issues
Critical

Box is cleared on every app startup — all local data is lost on relaunch

main.dart:93 calls await box.clear() unconditionally before the TODOLIST == null check at line 95. Because the box is always empty after clear, the condition is always true, createInitialData() is always called, and loadData() is never reached. Any task the user creates is gone the next time they open the app. 

// main.dart — current (broken)
var box = await Hive.openBox("mybox");
await box.clear();                           // ← WIPES everything
if (_myBox.get("TODOLIST") == null) {      // ← always null now
  db.createInitialData();
} else {
  db.loadData();                             // ← unreachable
}

// Fix: remove the clear() call
var box = await Hive.openBox("mybox");
if (box.get("TODOLIST") == null) {
  db.createInitialData();
  db.updateDataBase();
} else {
  db.loadData();
}
Critical

Task is a positional List<dynamic> with 18 magic indices

Every task is stored and accessed as an untyped list. There is no documentation of what each index means. Code all over the codebase uses raw literals like task[0], task[13], task[17]. A single field insertion anywhere in the list silently corrupts every piece of code that reads the list — with no compile-time error.

02 — High Severity
High

Everything in a single Hive box with string keys

All data — tasks, calendar events, categories, settings, account details, sync state — is crammed into one box named "mybox". Key collisions are one typo away, there is no namespacing, and you cannot open/close a box independently. Industry standard is one box per entity type (or use a proper embedded database).

High

loadData() duplicates the logic of all individual loadXxx() methods

The class has both loadData() (a combined loader) and separate loadToDoList(), loadGoogleCalTasks(), etc. The same parse-from-Hive code is written twice for every entity. Any bug fix must be applied in two places — and currently they are already out of sync: loadData() inline-reads the box while the individual methods call the box via the getter. They will diverge further over time.

High

updateDataBase() rewrites every key on every change

A single field update (e.g. marking one task complete) calls updateDataBase() which serialises and writes all tasks, all calendar events, all categories, settings, and account details in one pass. On a large task list this is unnecessary I/O. Each collection should be saved independently via its targeted save method.

High

Global singleton db passed via prop-drilling through every screen

main.dart declares final ToDoDataBase db = ToDoDataBase() at the top level. It is then passed as a constructor argument through TaskPage → MyDrawer → CalendarPage → StatisticsPage → CalenderSyncPage etc. There is a Provider system in place but db itself is not provided — making testing and widget composition difficult.

03 — Medium Severity
Medium

No data migration strategy

If you add a new field to the task schema (e.g. an index [18]), all existing stored tasks will be missing that slot. There is no version number in the stored data and no migration step. A schema change will silently produce RangeError on all existing installs.

Medium

Three parallel task lists with no shared schema enforcement

toDoList, googleCalTasks, localCalTasks, and outlookCalTasks are all List<List<dynamic>> but their index maps are different. SyncTile reads task[15] as the calendar event ID and task[17] as the source, which is calendar-task-specific. Mixing these up causes silent wrong-data bugs.

Medium

Timezone detection is half-implemented and silently falls back to UTC

createInitialData() has an empty if (settings["timeZone"] == "") { } block. initLocalTimeZone() in main.dart uses a brute-force loop over all timezone names matching the current UTC offset — which is wrong for half-hour offsets and ambiguous in DST transitions. The native timezone plugin is commented out. This means users in non-UTC zones can get silently incorrect reminder times.

Medium

accountDetails duplicates state already in AuthProvider

User name and profile picture are stored both in Hive (accountDetails) and in AuthProvider / GoogleAuthService.currentUser. These two sources can become inconsistent (e.g. user changes their Google profile name). There should be one authoritative source — the auth service — and Hive should only cache a token, not profile metadata.

04 — Low Severity / Code Quality
Low

23+ print() statements left in production code

database.dart and main.dart print full task lists, calendar event arrays, and debug state on every load and save cycle. In release builds this is dead code. It also leaks personal task data to device logs. Replace with a conditional logger (kDebugMode guard or the logger package).

Low

int a = 5; at the top of date_time_utils.dart

An unused global variable int a = 5 is declared at the module level. This is a stale debug artifact and should be deleted.

Low

Large blocks of commented-out code throughout

database.dart, main.dart, and task_page.dart contain dozens of commented-out lines (old task schema, disabled timezone plugin, old notification tests). These should live in git history, not in source.

07 — Prioritised Improvements
1

Remove box.clear() from startup 15 min Critical fix

Delete line 93 in main.dart. This is the single most impactful change — it restores persistence immediately without any other refactor.

2

Define a Task model class with a Hive TypeAdapter 2–4 hrs High impact

Replace List<dynamic> with a typed class. Hive's code-gen (hive_generator) produces a TypeAdapter automatically from annotations. All index-based access becomes named field access with compiler-checked types.

@HiveType(typeId: 0)
class Task extends HiveObject {
  @HiveField(0) late String  name;
  @HiveField(1) late bool    completed;
  @HiveField(2)       String? note;
  @HiveField(3)       String? dueDateUtc;   // yyyy-MM-dd
  @HiveField(4)       String? dueTimeUtc;   // HH:mm
  @HiveField(5) late String  category;
  @HiveField(6) late String  priority;      // or Priority enum
  @HiveField(7)       String? repeatType;
  @HiveField(8)       int?    reminderAmount;
  @HiveField(9)       String? reminderType;
  @HiveField(10) late bool   isStarred;     // was String "true"/"false"
  @HiveField(11) late String createdAt;
  @HiveField(12) late String id;
  @HiveField(13)       List<SubTask>? subtasks;
  @HiveField(14)       String? calendarName;
  @HiveField(15)       String? nativeEventId;
  @HiveField(16)       String? remoteEventId;
  @HiveField(17) late String source;         // google/outlook/local
}

Separate Hive boxes per entity type 1 hr High impact

Open one box per collection. This isolates failures, allows lazy loading, and makes intent clear.

4

Replace global singleton with a TaskProvider 3–5 hrs High impact

Register TaskProvider extends ChangeNotifier in MultiProvider alongside your existing providers. It holds the task list, calls Hive under the hood, and calls notifyListeners() on every mutation. All widgets consume it via context.watch<TaskProvider>() — no more prop-drilling. 

class TaskProvider extends ChangeNotifier {
  final _box = Hive.box<Task>('tasks');

  List<Task> get tasks => _box.values.toList();

  Future<void> addTask(Task t) async {
    await _box.put(t.id, t);
    notifyListeners();
  }

  Future<void> completeTask(String id) async {
    final t = _box.get(id);
    if (t == null) return;
    t.completed = true;
    await t.save();               // HiveObject.save() — only writes this record
    notifyListeners();
  }
}
5

Add a schema version + migration helper 2 hrs

Store a schemaVersion key in the settings box. On startup, compare it to the current app version. If lower, run the appropriate migration function before loading data. This makes future field additions safe. 

const int kCurrentSchema = 2;

void migrateIfNeeded() {
  final stored = settingsBox.get('schemaVersion', defaultValue: 1);
  if (stored < 2) _migrateV1ToV2();
  settingsBox.put('schemaVersion', kCurrentSchema);
}

Fix the timezone detection 30 min

Un-comment and re-add the flutter_timezone package (already in the commented imports). It calls the OS directly and returns the correct IANA name ("Asia/Colombo") without any offset-matching loop.
