1. Executive Summary

This audit evaluates the authentication and authorization pipeline of the Flutter To-Do app against industry-standard practices (OAuth 2.0 RFC 6749, OWASP MASVS, Google/Microsoft SDK best practices). The app uses OAuth 2.0 delegation exclusively — no password handling — which is fundamentally sound. However, several implementation gaps introduce security and reliability risks ranging from critical configuration mismatches to missing access controls.

2. Authentication Architecture Overview

The app integrates three separate identity providers and one backend service:

Provider Protocol Service File Scopes Requested Token Storage
Google OAuth 2.0 (google_sign_in SDK) google_sign.dart email, calendar, drive.file In-memory + SDK-managed (auto-refresh)
Microsoft OAuth 2.0 / MSAL (msal_auth SDK) outlook_sign.dart User.Read, Calendars.ReadWrite FlutterSecureStorage (key: outlook_cal_accessToken)
Supabase JWT / REST main.dart (init only) Anonymous (anonKey) Supabase SDK-managed
AI Endpoint REST (no auth) AiTaskService.dart None N/A

App Startup Auth Flow (main.dart lines 80–111)

// 1. Supabase initialized with hardcoded anonKey
Supabase.initialize(url: ..., anonKey: "eyJ...")

// 2. GoogleAuthService: silent then FORCED interactive sign-in
await GoogleAuthService.initApp();     // signInSilently() → signIn() if null

// 3. OutlookAuthService: silent then FORCED interactive sign-in
await OutlookAuthService.initialize();  // acquireTokenSilently() → signIn() if null

// 4. App launches with NO auth state provider in widget tree
runApp(MultiProvider(...))             // no AuthProvider included

Both Google and Outlook authentication are forced at startup before the user has interacted with calendar features. This is an unconventional UX pattern — industry standard is lazy/deferred auth triggered when the protected feature is first accessed.

4. Detailed Findings

Critical F-01 — MSAL Client ID Mismatch

outlook_sign.dart:5 · android/app/src/main/assets/msal_config.json

The client ID hardcoded in OutlookAuthService and the client ID in the MSAL Android configuration file are different values. MSAL uses the config file to validate the redirect URI against the registered app. When these differ, authentication either silently fails or silently authenticates against the wrong Azure AD application registration, potentially granting tokens scoped to a different application. 

// outlook_sign.dart:5
static const _clientId = '6450d522-3a1c-4005-ae93-1fdc7f91aea2';

// msal_config.json — different ID!
"client_id": "1046c7a4-3a52-4845-b477-81163b44f47e"
Fix: Decide which Azure AD app registration is the correct one and make both files reference the same client ID. Verify the redirect URI is registered for that app in Azure Portal under Authentication → Redirect URIs.

High F-02 — Forced Interactive Sign-In at Cold Start

main.dart:100–101 · google_sign.dart:24–25 · outlook_sign.dart:137–138

Both Google and Outlook authentication attempt interactive sign-in dialogs at app startup, before the user has navigated to any calendar feature. This violates the OAuth 2.0 principle of requesting only the permissions needed at the time they are needed ("incremental authorization"), and it means:

  • Users who only want the task manager (no calendar) are forced through two OAuth flows.
  • If either flow fails or is cancelled, the failure is silently swallowed.
  • Outlook's initialize() may open a full browser-based login at cold start.
// google_sign.dart — initApp() forces sign-in if silent fails
static Future<void> initApp() async {
  GoogleSignInAccount? user = await signInSilently();
  if (user == null) user = await signIn(); // ← interactive dialog on first launch
}
Fix: Replace initApp() with silent-only restoration at startup. Trigger interactive sign-in only when the user taps a calendar-sync button or navigates to /calendarSync. Use signInSilently() exclusively at startup.

High F-03 — No Route-Level Authorization Guards

main.dart:144–163

All seven application routes are defined in MaterialApp.routes with no authentication check. Any deep link or programmatic navigation can reach any page without an authenticated session. While this is a personal to-do app (single user), the pattern establishes no boundary between unauthenticated and protected state, which is especially relevant because the app stores and displays user data loaded from Hive on startup. 

// main.dart — no guard on any route
routes: {
  '/':                (context) => TaskPage(...),
  '/calendarSync':    (context) => CalenderSyncPage(...),  // needs auth
  '/statistics':      (context) => StatisticsPage(...),
  '/calendar':        (context) => CalendarPage(...),       // needs auth
  ...                                                         // no AuthGuard
}
Fix: Introduce an AuthProvider (ChangeNotifier) tracking sign-in state for each provider. In CalenderSyncPage and CalendarPage build methods, check Provider.of<AuthProvider>(context).isGoogleSignedIn and redirect to the sign-in page if false. Alternatively, migrate to go_router with redirect callbacks for a cleaner solution.

High F-04 — Access Token Logged to Console

outlook_sign.dart:79

A live access token is printed to the debug console. In debug builds this is visible in any attached debugger or log aggregation tool. If any crash reporting SDK (Crashlytics, Sentry) is added in the future, tokens may be transmitted to third-party servers. 

// outlook_sign.dart:79
print("🔑 Silent token acquired: ${result.accessToken}"); // ← full token in log

// outlook_sign.dart:112
print('Access Token: ${result.accessToken}'); // ← also in interactive sign-in
Fix: Remove all print() statements containing token values. If debug logging is needed, log only the first 8 characters (e.g., ${token.substring(0,8)}...) or use a conditional: if (kDebugMode) debugPrint(...) with a token hint only.

Medium F-05 — Unauthenticated AI / Supabase Edge Function Call

AiTaskService.dart:20–26

The AI task generation endpoint is called with only a Content-Type header and no authorization. Since Supabase Edge Functions can enforce JWT authentication, this endpoint is either publicly open (no row-level security possible) or relying solely on the secret function URL for security ("security by obscurity"). Neither is acceptable for a production API that may incur LLM costs per call. 

// AiTaskService.dart:22–25
final response = await http.post(
  Uri.parse(_endpoint),
  headers: {'Content-Type': 'application/json'}, // ← no Authorization header
  body: jsonEncode(body),
);
Fix: Add the Supabase anon key as a Bearer token header: 'Authorization': 'Bearer <anonKey>'. Better: use the Supabase Dart client's functions.invoke() which automatically attaches the session JWT. On the server side, enable verify_jwt: true in the function configuration.

Medium F-06 — Supabase anonKey Hardcoded in Source

main.dart:85–86

The Supabase anonymous key (a long-lived JWT) is hardcoded directly in main.dart and committed to the repository. While Supabase's anon key is intentionally public-facing (it's used client-side and protected by Row-Level Security), embedding it directly in source means it will appear in git history permanently, even if rotated. Anyone with read access to the repo has the key. 

// main.dart:85–86
anonKey: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc3MiOiJzdXBhYmFzZSIsIn...",
Fix: Move to a .env file (using flutter_dotenv or compile-time --dart-define flags) and add .env to .gitignore. Example: flutter run --dart-define=SUPABASE_ANON_KEY=eyJ... then read with const String.fromEnvironment('SUPABASE_ANON_KEY').

Medium F-07 — Fragmented Auth State (No AuthProvider)

google_sign.dart · outlook_sign.dart · data/database.dart

Authentication state is stored in three separate locations with no synchronization:

  • GoogleAuthService.currentUser — static nullable field (in-memory)
  • OutlookAuthService._accessToken — private static nullable string (in-memory)
  • db.accountDetails["userName"] — Hive-persisted user display info

This means widgets reading from different sources can display inconsistent state (e.g., the drawer showing a user name from Hive while currentUser is null after a token expiry). There is no ChangeNotifier to rebuild UI on auth state changes, so sign-out does not automatically update dependent widgets.

Fix: Create a single AuthProvider extends ChangeNotifier registered in MultiProvider. It should expose isGoogleSignedIn, isOutlookSignedIn, and currentUserDisplayName, and call notifyListeners() on sign-in/sign-out. All widgets should read from this single source.

Medium F-08 — Over-Privileged Google Calendar Scope

google_sign.dart:11

The app requests the full read/write https://www.googleapis.com/auth/calendar scope. Google's calendar API provides a read-only scope (calendar.readonly) and an event-only write scope (calendar.events). If any feature only needs to read events, the full write scope is unnecessarily broad, violating the principle of least privilege. 

// google_sign.dart:9–13
scopes: [
  'email',
  'https://www.googleapis.com/auth/calendar',  // ← full read+write+settings
  'https://www.googleapis.com/auth/drive.file', // ← this one is appropriately scoped
]
Fix: Audit which calendar operations are performed. If only event creation/editing is needed, replace with https://www.googleapis.com/auth/calendar.events. This also reduces the permissions shown in the Google consent dialog, increasing user trust.

Low F-09 — Incomplete Sign-Out (Outlook Token Not Deleted)

outlook_sign.dart:59–67

OutlookAuthService.signOut() calls _pca.signOut() and clears the in-memory _accessToken, but does not delete the persisted token from FlutterSecureStorage. On next app launch, restoreLastSession() will find and attempt to reuse this stale token. 

// outlook_sign.dart:59–67
static Future<void> signOut() async {
  await _pca.signOut();
  _accessToken = null;
  // ← missing: await storage.delete(key: 'outlook_cal_accessToken');
}
Fix: Add await storage.delete(key: 'outlook_cal_accessToken'); inside signOut().

Low F-10 — Account Details Not Cleared on Sign-Out

pages/sign_in_page.dart (sign-out handler)

The Google sign-out handler clears GoogleAuthService state but leaves db.accountDetails (Hive) populated with the previous user's name and profile picture URL. On next launch, the drawer will briefly show the stale user name before auth state is resolved.

Fix: In the sign-out handler, call db.accountDetails = {"userName": "none", "profilePicture": "none"}; db.saveAccountDetails(); to clear persisted display data alongside auth state.

5. Code-Level Bugs

Bug B-01 — Stale Token Written to Storage in initialize()

outlook_sign.dart:130–134

In OutlookAuthService.initialize(), the storage write uses _accessToken (the old/null value) instead of silentToken (the freshly acquired value). The in-memory assignment happens after the write. This means the stored token is always one session behind — or null on first run. 

// outlook_sign.dart:127–135 — current (buggy)
if (silentToken != null) {
  await storage.write(
    key: 'outlook_cal_accessToken',
    value: _accessToken,   // ← BUG: _accessToken is still the OLD value here
  );
  _accessToken = silentToken; // ← assignment happens AFTER the write
  return true;
}

// Fix: swap the order
if (silentToken != null) {
  _accessToken = silentToken;             // ← assign first
  await storage.write(
    key: 'outlook_cal_accessToken',
    value: silentToken,                   // ← then write the fresh value
  );
  return true;
}

Bug B-02 — restoreLastSession() Trusts Expired Token Before Verifying

outlook_sign.dart:32–33

restoreLastSession() assigns the stored token to _accessToken before calling acquireTokenSilently() to verify it. If acquireTokenSilently() fails (line 35), the function returns false but _accessToken is already set to the expired token. Any code path that calls OutlookAuthService.accessToken after a failed restoration will receive an expired token and make failing API calls with no indication of why. 

// outlook_sign.dart:32–56 — current (buggy)
_accessToken = savedToken;  // ← set BEFORE verifying
final silentToken = await acquireTokenSilently();
if (silentToken != null) { ... return true; }
// silentToken is null here — _accessToken still holds the expired token!
return false;  // ← but _accessToken was never cleared

// Fix: only assign after verification succeeds
final silentToken = await acquireTokenSilently();
if (silentToken != null) {
  _accessToken = silentToken; // ← assign only on success
  await storage.write(key: 'outlook_cal_accessToken', value: silentToken);
  return true;
}
_accessToken = null; // ← explicitly clear on failure
return false;

Bug B-03 — _GoogleAuthClient Inner Client Never Closed

google_sign.dart:106–113

_GoogleAuthClient creates a new http.Client() at instantiation. _initApis() creates a new _GoogleAuthClient on every call (every API use via ensureApisReady()), meaning each call leaks an HTTP connection pool. http.Client holds open connections and should be closed when no longer needed. 

// google_sign.dart:106–113 — current (leaky)
class _GoogleAuthClient extends http.BaseClient {
  final http.Client _inner = http.Client(); // ← new client every instantiation
  // close() is never called; previous client instances are discarded
}
Fix: Either (a) make _inner a static singleton, or (b) override close() in _GoogleAuthClient to call _inner.close(), and call calendarApi = null; client.close() on sign-out. Option (a) is simpler for this use case.
